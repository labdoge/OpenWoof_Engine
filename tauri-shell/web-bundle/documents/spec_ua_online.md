## 1. 什麼是輔助代理人 (UA)

### 一句話定義

**UA 是一個獨立於主敘事 LLM 的「幕後裁判」，負責判定每一回合的好感度變化、場景重要性、任務進度，讓主敘事 LLM 專心寫故事。**

### 為什麼需要 UA

在沒有 UA 的架構下，主敘事 LLM 必須同時扮演「說故事的人」和「記分的裁判」。這會導致：

- **分數膨脹**：LLM 傾向討好玩家，好感度增長過快，缺乏節奏
- **判斷不一致**：同樣的玩家行為在不同回合可能得到截然不同的評價
- **機制盲區**：LLM 無法可靠地追蹤情緒歷史、任務條件、關鍵事件

UA 將「評價」與「敘事」拆開，由一個專職的 LLM 呼叫（使用相同模型，但獨立的 prompt 與上下文）來處理所有機制判定，主 LLM 只需要讀取結果並據此調整角色行為。

### 核心原則

| 原則 | 說明 |
|------|------|
| **敘事歸主 LLM，數字歸 UA** | 主 LLM 不計算任何分數；UA 不撰寫任何敘事文字 |
| **客戶端權威** | 回合數、點數等純機械數據由前端計算，不由任何 LLM 決定 |
| **UA 權威** | 好感度變化、任務完成判定、場景評估由 UA 裁定，前端直接採用 |
| **單一 NPC** | 每個場景僅有一位主要 NPC，所有評估圍繞該 NPC 進行 |

## 2. UA 的核心職責

UA 在**每一回合**被呼叫一次，輸入為該回合的對話上下文，輸出為一份結構化的評估報告。

### 2.1 好感度變化 (Affection Delta)

**好感度是 NPC 對玩家的長期情感累積，範圍 0 ~ 100。**

UA 直接判定每回合的好感度變化，範圍 **-10 ~ +10**：

| 分數範圍 | 代表情境 | 附帶效果 |
|---------|---------|---------|
| -10 ~ -3 | 嚴重冒犯、背叛、觸碰底線 | **觸發關鍵事件**（|Δ| ≥ 3） |
| -2 ~ -1 | 不體貼、略失禮、忽視 NPC 感受 | — |
| 0 | 完全中性的互動 | — |
| +1 ~ +2 | 日常互動、普通對話、沒有特別觸動 | — |
| +3 ~ +5 | 有意義的互動——體貼、風趣、理解 NPC | **觸發關鍵事件** |
| +6 ~ +8 | 深刻的情感互動——守護、坦誠、挺身而出 | **觸發關鍵事件** |
| +9 ~ +10 | 極致時刻——僅出現於任務完成等重大突破 | **觸發關鍵事件** |

> **設計意圖**：好感度可以下降，但**永遠不會低於 0**。前端在套用 delta 時做 `max(0, current + delta)` 的下限保護。正常遊玩中大多落在 +1~+3，高分段（+6 以上）為罕見獎勵。負值段讓玩家的錯誤選擇有代價，但不會「倒扣到絕望」。

#### 任務對好感度的影響

- **任務完成**：固定獎勵 **+10**（最高值），玩家能明確感受到獎勵
- **有未完成的角色請求時**：好感度增長受到抑制（建議上限降為 1~5），鼓勵玩家關注 NPC 的請求

#### 好感度等級（供主 LLM 參考 NPC 行為基線）

| 等級 | 範圍 | NPC 行為基線 |
|------|------|------------|
| 警戒 | 0 ~ 20 | 防備、簡短回應 |
| 中立 | 20 ~ 40 | 禮貌、有距離感 |
| 友善 | 40 ~ 60 | 健談、偶爾主動幫忙 |
| 信任 | 60 ~ 80 | 開放、分享心事 |
| 摯愛 | 80 ~ 90 | 脆弱、尋求親近 |
| 獻身 | 90 ~ 100 | 犧牲自我利益 |

### 2.2 表情選擇 (Portrait Hint)

UA 每回合為 NPC 選擇最適合的表情，前端據此切換立繪：

| 編號 | 標籤 | 適用情境 |
|------|------|---------|
| 0 | default | 真正中性、無情緒波動的時刻 |
| 1 | happy | 開心、有趣、輕鬆 |
| 2 | surprised | 驚訝、震驚、意外 |
| 3 | sad | 難過、沮喪、失落 |
| 4 | angry | 生氣、不悅、惱怒 |

**注意**：表情反映 NPC 的**外顯情緒**，不一定和好感度變化一致。NPC 可能因為被調侃而 `angry` 但好感度實際上升，或因為聽到壞消息而 `sad` 但和玩家的關係完全無關。

### 2.3 內心獨白 (Inner Thought)

每回合 UA 輸出 NPC 的內心想法，顯示於對話氣泡下方的常駐氣泡。內心獨白**同時傳達情緒與想法**：

**情緒層**：NPC 當下的情感狀態
> 「（心跳好快...他怎麼能這麼自然地說出這種話）」

**想法層**：NPC 對當前情境的思考或反應
> 「（這個人好奇怪，但...不討厭）」

**期望層**（有時）：NPC 內心渴望的事
> 「（好希望他能安慰我一下...）」

> 內心獨白是玩家理解 NPC 的重要窗口。它讓玩家看見 NPC 的「真心話」，增加與角色的情感連結。即使玩家不執行任何任務，單純閱讀內心獨白也是一種享受。

### 2.4 關鍵事件偵測 (Critical Events)

**好感度變化 ≥ 3 時，UA 自動標記一則關鍵事件。**

關鍵事件是一句精簡的回顧，記錄「發生了什麼、結果是什麼」：

> 「玩家主動替小雅擋下母親的追問，小雅第一次露出感激的表情。」

關鍵事件的用途：
- **永久記錄**：存入資料庫，跨場景持續存在，成為角色關係史的里程碑
- **上下文摘要**：壓縮對話歷史時，關鍵事件被優先保留
- **任務完成事件**：任務完成也會自動生成一則關鍵事件（帶特殊標記）

### 2.5 場景狀態擷取 (Status Extraction)

UA 從對話中提取當前場景的關鍵視覺資訊，供狀態面板顯示：

- **地點**：當前場景位置（≤15字）
- **時段**：清晨 / 上午 / 午後 / 傍晚 / 夜晚 / 深夜
- **NPC 情緒**：一句話內心描述（≤40字，第一人稱）
- **NPC 外觀**：當前服裝與外貌特徵
- **場景氛圍**：2~4 字的感官描述（如「年夜飯香氣」「雨聲淅瀝」）

> 好感度數值、回合數等機械數據由前端管理，不由 UA 擷取。

## 3. 場景評估與敘事導引

### 3.1 場景重要性分級

UA 為每回合標記重要性等級。**此分級為建議性質**——線上版已透過輸出格式控制內容長度，場景分級的主要用途是**協助 UA 判定好感度變化的合理範圍**。

#### 日常 (Routine)
- 普通閒聊、例行動作
- 沒有情感波動或劇情推進
- → 好感度建議範圍：1 ~ 2

#### 值得注意 (Notable)
符合以下**至少一項**：
- 有意義的情感互動（NPC 有明確情緒反應）
- 任務有進展或出現相關線索
- 首次出現的親密或脆弱時刻
- 新的重要資訊揭露
- → 好感度建議範圍：2 ~ 5

#### 高潮 (Climactic)
**嚴格條件**——不確定時歸類為「值得注意」：
- 關係狀態永久改變
- 重大秘密首次揭露
- 任務完成
- 角色做出不可逆的決定
- → 好感度建議範圍：5 ~ 8（任務完成固定 10）

**冷卻機制**：高潮回合之後，下一回合自然回落到「值得注意」或「日常」。

### 3.2 敘事導引機制

UA 不直接撰寫故事，但透過以下方式間接引導敘事方向：

#### 好感度等級 → NPC 行為基線
主 LLM 根據當前好感度等級，調整 NPC 的態度和行為。好感度越高，NPC 越主動、越坦率。

#### 場景重要性 → 好感度校準
場景分級幫助 UA 確保好感度變化幅度與場景實際重要性匹配，防止日常閒聊給出過高分數。

#### 內心獨白 → 玩家引導
NPC 內心獨白為玩家提供「NPC 在想什麼」的軟性提示，但不強制。玩家完全可以忽略，自由互動。

#### 劇本指令導引 → 主線推進
**當場景處於日常狀態，且沒有正在進行中的關鍵互動時**，UA 應引導主 LLM 在敘事中**自然地引入或暗示劇本指令的相關情境**。

這種引導必須是**隱微的**——不是直接告訴玩家「去做 X」，而是讓場景自然地朝向任務觸發方向發展：

> 範例：劇本指令是「幫 NPC 應對母親的結婚追問」。UA 在日常回合可以引導主 LLM 讓母親在遠處提起相關話題，或讓 NPC 露出擔憂的表情——為玩家創造介入的機會，但不強制。

## 4. 任務系統：角色請求與劇本指令

### 設計理念

任務系統是一個**增添趣味的獎勵機制**——它給玩家明確的互動目標，讓好感度的成長有「突破性時刻」。任務不是必要的遊戲進程，而是讓體驗更有層次的調味料。

任務分為兩種類型：

| | 角色請求 (NPC Quest) | 劇本指令 (Scenario Quest) |
|---|---|---|
| **驅動源** | NPC 的個性、情緒、與玩家的關係 | 場景劇本的主線劇情 |
| **來源** | UA 設計方向 → 主 LLM 生成完整描述 | 內容創作者預寫於場景 JSON → UA 充實細節 |
| **強制性** | 完全可選，但會影響好感度增長速度 | 完全可選，如同沙盒遊戲的主線任務 |
| **同時上限** | 1 個 | 1 個 |
| **完成獎勵** | 好感度 +10（最高值）+ 關鍵事件 | 好感度 +10（最高值）+ 關鍵事件 |

### 4.1 角色請求 (NPC Quest)

#### 什麼是角色請求

角色請求是 **NPC 基於自身個性和當前情境，對玩家產生的具體期望**。它由 UA 決定方向，再由主 LLM 轉化為具體的任務描述。

> 範例：
> - 小雅被客人騷擾後 → UA 判定：NPC 需要保護 → 主 LLM 生成任務：「在下一次有人騷擾小雅時替她出頭」
> - 凌風心情低落 → UA 判定：NPC 需要傾聽 → 主 LLM 生成任務：「和凌風聊聊他最近的煩惱」

#### 觸發頻率

- **最早觸發**：第 3 回合起（給玩家時間熟悉角色和場景）
- **完成後冷卻**：距上一次任務完成至少 3 回合
- **最大空窗期**：若超過 5 回合沒有角色請求，UA 應主動產生一個
- **阻塞規則**：同時只能有 1 個角色請求。未完成的請求會持續存在，阻止新請求產生

#### 任務結構（簡潔直接）

每個角色請求包含：

- **標題**：2~6 字，概括任務（如「安慰之言」「信任考驗」）
- **描述**：一句話說明 NPC 的期望（≤40 字）
- **條件**：**恰好 2 個**，每個條件描述一個具體的玩家行為（≤60 字）

```
標題：安慰之言
描述：小雅被母親的話傷到了，希望有人能理解她
條件：
  1. 在私下與小雅談論剛才的事情
  2. 表達對她處境的理解而非敷衍安慰
```

#### 條件設計原則（針對 Flash 3.0 / 3.1 優化）

- **恰好 2 個條件**：不多不少，降低判定複雜度
- **玩家為主詞**：只描述玩家做什麼，不描述 NPC 反應
- **動作明確**：使用具體動詞（「主動提起」「替她擋下」「坦白說出」），避免模糊形容（「展現真心」「互動良好」）
- **可觀測**：條件必須能從對話文字中直接判斷，不依賴推測

#### 生成流程

```
UA 判定：該產生角色請求了
    ↓
UA 輸出任務方向（questDirective）：
  { npcId, theme, briefDirection, affectionContext }
    ↓
主 LLM 根據方向 + 對話上下文生成完整任務：
  { title, description, conditions[2] }
    ↓
前端接收 → 觸發任務通知 → 記錄為進行中
```

> **為什麼讓主 LLM 生成？** UA 專注於「什麼時候該有任務」和「任務大方向」的判斷。主 LLM 更擅長根據對話語境撰寫貼合角色個性的任務文案。分工讓兩者各司其職。

#### 好感度抑制

**當有未完成的角色請求時，正常好感度增長上限降為 1~3。** 這鼓勵玩家關注 NPC 的期望，而非讓好感度在忽略請求的情況下持續快速增長。

任務完成瞬間的 +10 獎勵不受此限制，形成「壓抑→釋放」的獎勵節奏。

### 4.2 劇本指令 (Scenario Quest)

#### 什麼是劇本指令

劇本指令是**由內容創作者預先編寫在場景 JSON 中的主線任務**，代表場景故事的核心脈絡。它是場景的「主線劇情」——玩家知道它存在，可以選擇推進或忽略。

> 範例（場景：「租借合約——假裝情侶回老家過年」）：
> 1. 在親戚面前成功演出一次「情侶互動」
> 2. 幫 NPC 應對母親關於「結婚」的追問
> 3. 在 NPC 被舊事觸動情緒時安慰她
> 4. 和 NPC 坦誠這段關係的真實感受

#### 設計特性

- **線性排序**：指令有先後順序，完成前一個才啟動下一個
- **完全可選**：玩家不推進主線也完全不影響自由互動
- **進度可見**：玩家能看到「主線進度 2/4」，但未啟動的指令內容隱藏
- **阻塞規則**：同時只能有 1 個劇本指令在進行中

#### 場景 JSON 定義

```json
{
  "scenario_directives": [
    {
      "order": 1,
      "direction": "在親戚面前成功演出一次情侶互動",
      "context_hint": "過年聚餐場景，長輩在場時"
    },
    {
      "order": 2,
      "direction": "幫 NPC 應對母親的結婚追問",
      "context_hint": "母親與 NPC 獨處或飯桌上提起婚事時"
    }
  ]
}
```

- `direction`：簡短的任務方向（由內容創作者撰寫）
- `context_hint`：觸發時機的提示（幫助 UA 判斷何時啟動）

#### 擴展與生成流程

```
UA 判定：劇情已推進到適合啟動下一個指令的時機
    ↓
UA 輸出啟動信號 + 充實後的任務方向：
  { order, expandedDirection, contextForGeneration }
    ↓
主 LLM 根據方向 + 對話上下文生成完整任務：
  { title, description, conditions[2] }
    ↓
前端接收 → 觸發任務通知 → 記錄為進行中 → 更新進度顯示
```

條件設計原則與角色請求相同：恰好 2 個、玩家為主詞、動作明確、可觀測。

#### 主線導引

當場景處於日常狀態（非高潮、沒有正在進行的關鍵互動），UA 應引導主 LLM **自然地將場景朝向下一個指令的觸發方向推進**：

- 不是直接告訴玩家「去做 X」
- 而是讓 NPC 行為、環境變化、對話話題自然地靠近指令的 `context_hint`
- 玩家可以順勢推進，也可以選擇做自己的事

### 4.3 完成判定

UA 統一負責判定任務完成，規則對兩種任務相同：

**必須全部符合**：
1. 玩家的行為**直接對應**任務目標
2. 2 個條件**都已滿足**（不是部分）
3. 任務話題在對話中**明確出現**

**以下情況不算完成**：
- 玩家態度真誠但聊的是無關話題
- 只滿足了其中一個條件
- NPC 反應正面但條件尚未觸發
- **不確定時，一律不算完成**——寧可多等一回合

#### 完成時觸發的事件

1. **好感度 +10**（最高值）
2. **關鍵事件**：自動生成一則帶任務標記的關鍵事件
3. **通知事件**：UA 發出任務完成信號，前端顯示完成通知（覆蓋式通知或醒目 UI 效果）
4. 劇本指令完成後：更新進度顯示，解鎖下一個指令（但不立即啟動）

### 4.4 任務通知系統

UA 在以下時機發出通知信號，前端根據信號顯示覆蓋式通知或醒目 UI 效果：

| 事件 | 信號 | 建議 UI |
|------|------|--------|
| 新角色請求產生 | `questEvent: "npc_quest_received"` | 浮動通知 + 任務日誌按鈕亮起 |
| 新劇本指令啟動 | `questEvent: "scenario_quest_activated"` | 浮動通知 + 進度更新 |
| 任務完成 | `questEvent: "quest_completed"` | 醒目完成動畫 + 好感度跳動 |
| 主線全部完成 | `questEvent: "scenario_complete"` | 特殊完結通知 |

### 4.5 任務 UI：任務日誌

任務系統透過獨立的**任務日誌 (Quest Journal)** 面板呈現，由一個常駐的日誌按鈕開啟（Modal UI）。

日誌內容包含：
- **角色請求區**：顯示當前進行中的角色請求（若有），含標題、描述、條件
- **劇本指令區**：顯示主線進度（如「2/4」），當前指令的標題與描述，未啟動的指令顯示為鎖定狀態
- **已完成區**：歷史任務的簡要列表

> **與內心獨白的關係**：任務首次產生時可能透過內心獨白暗示（如「好希望有人能...」），但之後任務的追蹤完全在日誌面板中進行。兩個系統可以獨立運作。具體的首次呈現方式待與設計師討論。

## 5. UA 與主對話的分工

### 架構總覽

```
┌──────────────────────────────────────┐
│              前端 (Client)            │
│  · 回合數、點數（客戶端權威）          │
│  · UI 渲染、狀態面板、任務日誌         │
│  · 通知系統                           │
└───────┬──────────────────┬───────────┘
        │ ① 對話上下文      │ ⑤ 敘事回應
        ▼                  ▼
┌───────────────┐    ┌───────────────┐
│   UA (評估)    │    │  主 LLM (敘事) │
│               │    │               │
│ · 好感度 Δ    │──→ │ · 讀取好感度   │
│ · 表情選擇    │    │ · 調整 NPC 行為│
│ · 場景分級    │    │ · 撰寫回應     │
│ · 內心獨白    │    │               │
│ · 關鍵事件    │    │ ③ 任務方向     │
│ · 任務判定    │──→ │ → 生成任務文案 │
│ · 狀態擷取    │    │               │
│ · 主線導引    │    │ · 融入導引     │
│ · 壓縮摘要    │    │               │
└───────────────┘    └───────────────┘
   ② 結構化 JSON         ④ 注入 UA 結果
        │                    ▲
        └────────────────────┘
```

### 每回合流程

1. **玩家輸入** → 前端傳送對話上下文
2. **UA 呼叫** → 傳入：對話歷史 + NPC 狀態 + 進行中任務。輸出：結構化 JSON
3. **前端處理** → 更新好感度、表情、狀態面板、處理通知事件
4. **主 LLM 呼叫** → 傳入：對話歷史 + UA 的好感度/場景分級/任務方向。輸出：敘事回應（若有任務方向，同時輸出任務文案）
5. **前端渲染** → 顯示 NPC 回應、更新內心獨白、處理任務更新

### UA 設計方向 → 主 LLM 生成

**UA 負責「何時」與「什麼方向」，主 LLM 負責「怎麼寫」。**

這種分工的好處：
- UA 的 prompt 保持精簡，專注於規則判定（適合 Flash 系列模型的嚴格性）
- 主 LLM 已經有完整的對話上下文和角色設定，更適合撰寫貼合語境的任務文案
- 減少 UA 的 token 消耗（只輸出方向性指示，不需要生成完整文案）

### 簡化架構的優勢

線上版不含戰鬥系統、骰子引擎、多 NPC 互動等複雜模組：

- **實作複雜度低**：只需維護兩個 prompt（UA prompt + 主 LLM prompt）
- **token 消耗可控**：UA 輸出為精簡 JSON（約 200~400 tokens）
- **模型統一**：UA 和主 LLM 使用同一系列模型（Gemini Flash），無需管理多模型配置

## 6. 技術參考

> 本節僅供工程團隊參考，非決策必讀。

### 6.1 UA 輸出結構

```jsonc
{
  // ── 好感度評估 ──
  "affectionDelta": 3,              // -10 ~ +10, floor at 0
  "sceneImportance": "notable",     // routine / notable / climactic
  "portraitHint": 1,                // 0=default, 1=happy, 2=surprised, 3=sad, 4=angry
  "innerThought": "（他剛才說的話好溫柔...心跳好快）",

  // ── 狀態擷取 ──
  "statusExtract": {
    "location": "老家客廳",
    "time": "傍晚",
    "locationNote": "年夜飯香氣",
    "npc": {
      "mood": "裝作自然但心裡很緊張",
      "appearance": ["紅棕長髮", "白框眼鏡"],
      "outfit": ["白色高領連身裙", "駝色西裝外套"]
    }
  },

  // ── 時間推進 ──
  "timeAdvancement": 0,

  // ── 關鍵事件（affectionDelta ≥ 3 時出現）──
  "criticalEvent": {                // 條件性
    "description": "玩家主動替小雅擋下母親的追問，小雅第一次露出感激的表情。"
  },

  // ── 任務方向（UA 判定該產生時）──
  "questDirective": {               // 條件性
    "type": "npc_quest",            // npc_quest 或 scenario_quest
    "theme": "情感支持",
    "briefDirection": "NPC 被母親的話傷到，需要玩家的理解和安慰",
    "affectionContext": "friendly"   // 當前好感度等級
  },

  // ── 劇本指令啟動（時機到時）──
  "activateDirective": {            // 條件性
    "order": 2,
    "expandedDirection": "母親即將提起結婚話題，NPC 需要玩家幫忙應對",
    "contextForGeneration": "母親剛提到鄰居家的孩子結婚了"
  },

  // ── 任務完成判定 ──
  "questCompletion": {              // 條件性
    "completed": true,
    "questId": "nq_001",
    "reason": "玩家在私下傾聽小雅的煩惱並表達理解"
  },

  // ── 任務通知 ──
  "questEvent": "quest_completed",  // 條件性

  // ── 主線導引（日常場景時）──
  "narrativeGuidance": "母親在廚房提到鄰居喜事，可自然帶入婚事話題",  // 條件性

  // ── 好感度抑制標記 ──
  "affectionSuppressed": true       // 有未完成角色請求時為 true
}
```

### 6.2 主 LLM 注入格式

UA 結果以系統訊息注入主 LLM：

```
[SYSTEM: NPC State]
好感度: 友善 (48%) | 本回合 Δ+3 | 表情: happy
場景: notable
進行中任務: 「化解婚事追問」(劇本指令 #2)
敘事導引: 母親在廚房提到鄰居喜事，可自然帶入婚事話題
[End State]
```

若 UA 發出 `questDirective`，主 LLM 同時收到：

```
[SYSTEM: Generate Quest]
類型: npc_quest
方向: NPC 被母親的話傷到，需要玩家的理解和安慰
好感度背景: friendly
請生成: 標題(2-6字) + 描述(≤40字) + 條件(恰好2個, 各≤60字)
[End Quest Request]
```

### 6.4 上下文分配建議

關於劇本指令在 prompt 中的配置：

- **快取區（cached prompt / system prompt）**：場景概要、NPC 設定、UA 規則、所有劇本指令的 `direction` 與 `context_hint` 列表
- **動態區（每回合注入）**：當前好感度、回合數、進行中任務的完整描述、最近 N 回合對話、壓縮摘要

## 7. 待討論事項

### 設計層面

**Q1：任務首次呈現方式**
任務日誌是獨立 Modal，但新任務產生時需要引起玩家注意。方案：

- A：內心獨白氣泡暗示一次 + 浮動通知 + 日誌按鈕亮起
- B：全螢幕任務卡片動畫（類似手遊的任務解鎖）
- C：聊天區域內嵌的系統訊息卡片

→ 待與設計師討論

**Q2：劇本指令的隱藏格式**
未啟動的劇本指令在日誌中顯示為鎖定狀態。鎖定時應該顯示什麼？

- A：完全隱藏，只顯示「??? / 4」
- B：顯示模糊的主題提示（如「與母親有關的挑戰」）
- C：僅顯示序號和鎖頭圖示

**Q3：好感度抑制的幅度**
當前設計為有未完成角色請求時上限降為 1~3。是否需要更精細的控制？

- 固定上限 3 足夠簡單，但可能在所有場景都感覺一樣
- 可選方案：根據角色請求的等待回合數漸進抑制（剛產生時不抑制，3 回合後開始）

**Q4：角色請求的 5 回合強制產生**
如果超過 5 回合沒有角色請求，UA 強制產生一個。但如果此時劇情正處於劇本指令的關鍵推進期，是否應該暫緩？

- 建議：劇本指令進行中時，角色請求的最大空窗期可延長至 8 回合

**Q5：主線全部完成後**
所有劇本指令完成後，場景如何繼續？

- A：顯示「主線完成」標記，之後只有角色請求
- B：無特殊處理，自由互動繼續
- C：觸發一個「尾聲」特殊任務

### 技術層面

**Q6：UA 與主 LLM 的呼叫順序**
當前設計為 UA 先行、主 LLM 後行（串列）。是否考慮並行呼叫以降低延遲？

- 串列優勢：主 LLM 能直接使用 UA 結果，任務生成更準確
- 並行優勢：回應速度更快
- 建議：維持串列——UA 的結果直接影響主 LLM 的行為，並行會導致結果不一致

**Q7：任務方向由 UA 設計、文案由主 LLM 生成的實作方式**
兩種可行方案：

- A：UA 呼叫回來後，在主 LLM 的 system prompt 中注入任務生成指示，主 LLM 在回覆中同時輸出敘事 + 任務文案（一次呼叫）
- B：UA → 主 LLM 生成任務文案 → 再呼叫主 LLM 生成敘事（兩次呼叫）

→ 建議方案 A 以節省一次 API 呼叫
